# PVE Center

Дашборд для мониторинга кластеров и отдельных хостов Proxmox VE. Десктопное приложение на PySide6.

© 2026 Taurus McLure — taurus@mclure.ru

## Возможности

- **Дерево объектов** — кластеры, хосты, пулы, ВМ/CT, хранилища с иконками и цветовой индикацией статуса
- **Мониторинг** — графики ЦП, RAM, сети, диска (RRD данные из PVE)
- **Оборудование ВМ** — конфигурация, параметры, история задач
- **Пулы** — сводка по пулам ВМ с прогресс-барами ресурсов
- **Хранилища** — сводка по всем storage, детализация по нодам, график заполнения
- **Содержимое storage** — резервные копии, диски ВМ, ISO, шаблоны (динамические вкладки)
- **Сеть, сервисы, диски хоста** — физические интерфейсы, PVE-сервисы, диски с дедупликацией FC multipath
- **Снапшоты ВМ** — все снапшоты на хосте в одной таблице
- **Статусные индикаторы** — иконки с цветными кружками (зелёный/красный/жёлтый)
- **Автообновление** — фоновый soft refresh каждые 20с, без потери выделения и вкладок
- **Уведомления** — всплывающие тосты при изменении статуса хоста или ВМ
- **Управление ВМ** — кнопки Старт/Выкл/Перезагр/Сброс/Стоп для QEMU и LXC
- **Управление хостами** — ПКМ по хосту: удалить (с удалением токена с сервера), пересоздать токен
- **Автосоздание токена** — диалог добавления сервера создаёт API-токен для указанного пользователя (аудит в PVE показывает реального оператора)
- **Шифрование токенов** — мастер-пароль + PBKDF2 (600k итераций) + Fernet (AES-256)
- **Динамические кластеры** — одна запись в конфиге, все ноды подтягиваются через `/cluster/resources`
- **Обработка ошибок** — недоступный хост отображается с диагнозом (DNS/таймаут/авторизация/...)

## Требования

- Python 3.12+
- Proxmox VE (кластер или отдельный хост)
- Доступ по API к PVE-хосту (порт 8006)

## Установка

```bash
git clone https://github.com/mcluremail/pvecenter.git
cd pvecenter
python -m venv venv

pip install PySide6 proxmoxer requests pyqtgraph cryptography
```

## Настройка

**Ручной способ** — создать `nodes.json` в корне проекта:

```json
[
  {
    "name": "pve01.example.local",
    "host": "pve01.example.local",
    "cluster": "production",
    "user": "pvecenter@pve",
    "token_name": "pvecenter-abc123",
    "token_value": "секретный-токен"
  },
  {
    "name": "hv01.example.local",
    "host": "hv01.example.local",
    "user": "pvecenter@pve",
    "token_name": "pvecenter-def456",
    "token_value": "секретный-токен"
  }
]
```

Параметры конфига:

| Поле | Описание |
|------|----------|
| `name` | Отображаемое имя (FQDN) |
| `host` | Адрес для подключения к PVE API |
| `cluster` | Имя кластера (для группировки). Если не указан — хост попадает в "Отдельные хосты" |
| `user` | Пользователь PVE (`root@pam`, `user@ipa`, ...) |
| `token_name` | ID токена |
| `token_value` | Значение токена |
| `skip` | `true` — пропустить при загрузке (для дубликатов, обслуживаемых кластерной записью) |

**Через интерфейс** — нажать `[+]` на папке "Кластеры" или "Отдельные хосты" в дереве, ввести хост + учётные данные, токен создастся автоматически. Диалог предзаполняет настройки в зависимости от выбранной папки (для кластеров включает флаг представительства).

### Кластеры

Для кластера достаточно **одной записи** — главной ноды с `cluster`. Остальные ноды подтягиваются динамически через `/cluster/resources`. Ноды кластера отображаются как `short@cluster` (например `pve02@production`).

### Шифрование

При первом запуске с токенами в `nodes.json` будет предложено:
1. Установить мастер-пароль
2. Зашифровать токены в `nodes.enc`
3. Удалить незашифрованный `nodes.json`

При последующих запусках — диалог ввода мастер-пароля. `nodes.enc` можно коммитить в git (токены защищены).
