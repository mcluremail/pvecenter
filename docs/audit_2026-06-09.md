## 🔴 Критические баги

### БАГ-1: `finished` эмитируется дважды — `StorageContentListWorker` и `StorageBackupWorker`

**Файл:** `ui/api/metrics.py`, строки 133–142 и 179–188

```python
finally:
    session.close()
    try:
        self.signals.finished.emit()   # ← первый раз
    except RuntimeError:
        pass
    try:
        self.signals.finished.emit()   # ← второй раз — copy-paste ошибка!
    except RuntimeError:
        pass
```

Оба воркера эмитируют `finished` дважды в одном `finally`. Если к `finished` подключён `_discard_worker` — двойной дискард безвреден (`set.discard` идемпотентен). Но любой другой слот сработает дважды. Copy-paste артефакт.

### БАГ-2: `StorageContentListWorker` и `StorageBackupWorker` эмитируют несуществующий сигнал `error_occurred`

**Файл:** `ui/api/metrics.py`, строки 130 и 176

```python
# ContentListSignals определяет:
class ContentListSignals(_FinishedMixin):
    result = Signal(str, str, list)
    error = Signal(str, str, str)    # ← правильный сигнал для ошибок
    # error_occurred НЕ объявлен

# Но в run() при общем Exception:
self.signals.error_occurred.emit(str(e))  # ← AttributeError!
```

`ContentListSignals` не имеет атрибута `error_occurred`. При любой ошибке (кроме `ReadTimeout`) выбрасывается `AttributeError`. Он **не** ловится `except RuntimeError`, поэтому распространяется в `finally`, тихо гасится. Реальные ошибки сети/прав молча проглатываются — UI ничего не получает.

**Правильные вызовы:**
- `StorageContentListWorker` → `self.signals.error.emit(self.storage_name, self.content_type, str(e))`
- `StorageBackupWorker` (строка 176) → `self.signals.backups_error.emit(self.storage_name, str(e))`

### БАГ-3: `update_data(final=True)` никогда не вызывается

**Файл:** `ui/mainwindow.py`, строки 444–446 и 528–535

```python
worker.signals.result_ready.connect(
    lambda data, w=worker, g=refresh_gen: (
        self.on_worker_finished(data, w, g),  # ← вызывается ПЕРВЫМ
        self._discard_worker(w)               # ← вызывается ВТОРЫМ
    )
)
```

Внутри `on_worker_finished`:
```python
fetch_workers = [w for w in self._workers if isinstance(w, FetchWorker)]
if not fetch_workers:
    self.tree_panel.update_data(..., final=True)  # ← никогда не достигается
```

В момент проверки воркер `w` ещё **находится в `_workers`** — он удаляется только после возврата из `on_worker_finished`. Поэтому `fetch_workers` всегда непустой при последнем воркере, `if not fetch_workers` всегда False, `_spin_timer` дерева (`TreePanel._spin_timer`) никогда не останавливается после загрузки.

**Исправление:** убрать `_discard_worker` из лямбды `result_ready` — `finished` уже подключён к `_discard_worker` через `_run_worker`:
```python
worker.signals.result_ready.connect(
    lambda data, w=worker, g=refresh_gen: self.on_worker_finished(data, w, g)
)
```

### БАГ-4: `all_pools` не очищается при `refresh_data`

**Файл:** `ui/mainwindow.py`, строки 427–432

```python
self.all_nodes.clear()
self.all_vms.clear()
self.all_storages.clear()
self.all_iso_images.clear()

# ← self.all_pools НЕ очищается!
```

После удаления пула в PVE → `refresh_data` → пул по-прежнему в `all_pools` → показывается в диалоге создания ВМ. Пулы накапливаются бесконечно при повторных обновлениях.

**Исправление:** добавить `self.all_pools.clear()` в блок очистки.

### БАГ-5: `_tasks_started` не сбрасывается при `refresh_data`

**Файл:** `ui/mainwindow.py`, строки 109, 406 (начало `refresh_data`), 524–526

```python
# __init__:
self._tasks_started = False

# refresh_data() — НЕ сбрасывает _tasks_started!

# on_worker_finished:
if not self._tasks_started:
    self._tasks_started = True
    QTimer.singleShot(0, self.refresh_cluster_tasks)
```

После первого запуска `_tasks_started = True`. При повторном `refresh_data()` (добавление хоста, обновление токена) немедленный старт задач больше не происходит. Задачи обновятся только через 60 секунд по таймеру.

**Исправление:** добавить `self._tasks_started = False` в начало `refresh_data()`.

## 🟠 Архитектурные проблемы

### ARCH-1: `_migrate_if_needed` может попытаться удалить директорию пакета

**Файл:** `config.py:35-38`

```python
if os.path.isdir(_base_dir()) and not os.listdir(_base_dir()):
    os.rmdir(_base_dir())   # ← _base_dir() это pve_center/ !
```

В практике невозможно (директория всегда содержит `.py` файлы), но логически неверно — `_base_dir()` возвращает директорию самого пакета.

---

### ARCH-2: `traceback.print_exc()` вместо `logger.exception()` — более 10 мест

**Файлы:** `backend.py` (строки 226, 255, 385, 521, 574, 625, 672, 763, 1065, 1110), `ui/api/metrics.py` (строки 80, 128, 174, 221, 265, 309, 378, 471, 546, 606)

Ошибки уходят в stderr, минуя logging-систему. При использовании файлового лога (`logging.FileHandler`) эти ошибки будут потеряны.

---

### ARCH-3: Дублирование кода рендеринга пулов/VM в `_build_tree`

**Файл:** `ui/tree_panel.py:443-464, 497-517`

Блоки рендеринга пулов и VM идентичны для кластерных и standalone-нод. Отличаются только переменной родительского элемента. Стоит вынести в отдельный метод `_render_vms_to_parent(parent_item, vms)`.

---

### ARCH-4: Хрупкая эвристика определения типа storage в `_on_item_clicked`

**Файл:** `ui/tree_panel.py:770`

```python
if val and (val.count(".") > 0 or "/" in val):
    data["host_name"] = val   # standalone хост
else:
    data["cluster"] = val     # кластер
```

Кластер с именем `"my.cluster"` определится как хост. Standalone-хост `"proxmox"` (без точек) определится как кластер. Нужно хранить тип явно в `ITEM_KEY_ROLE` кортеже, а не угадывать по имени.
